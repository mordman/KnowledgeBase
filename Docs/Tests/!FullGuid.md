# 📘 Полный гид по видам тестирования

## 📊 Общая схема классификации

```
                        ВИДЫ ТЕСТИРОВАНИЯ
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
   ПО УРОВНЮ ДОСТУПА      ПО УРОВНЯМ            ПО ЦЕЛЯМ/
   (Box Testing)          ТЕСТИРОВАНИЯ            ХАРАКТЕРУ
        │                       │                       │
   ┌────┴────┐            ┌─────┴─────┐          ┌─────┴─────┐
   │         │            │           │          │           │

   │         │            │           │          │           │
 Gray      Glass       System    Acceptance   Positive    Performance
 Box       Box          │           │          │           │
                     Regression  Smoke      Negative   Security
                                                        Compatibility
```

## 1️⃣ ПО УРОВНЮ ДОСТУПА К КОДУ (BOX TESTING)

### ⬛ Black Box Testing (Тестирование чёрного ящика)
**Суть:** Тестирование без знания внутренней структуры кода. Проверка входных и выходных данных.

**Когда применять:**
- Приемочное тестирование
- Функциональное тестирование
- Системное тестирование

**Пример тест-кейса:**
```
ID: BB-001
Название: Авторизация с валидными данными
Предусловия: Пользователь зарегистрирован
Шаги:
1. Открыть страницу логина
2. Ввести email: user@test.com
3. Ввести пароль: ValidPass123
4. Нажать "Войти"
Ожидаемый результат: Переход в личный кабинет, отображение имени пользователя
```

---

### ⬜ White Box Testing (Тестирование белого ящика)
**Суть:** Тестирование с полным знанием внутренней структуры, логики и кода приложения.

**Когда применять:**
- Unit-тестирование
- Интеграционное тестирование
- Проверка покрытия кода

**Пример тест-кейса:**
```
ID: WB-001
Название: Проверка ветвления в методе calculateDiscount()
Код для проверки:
  if (amount > 1000) return amount * 0.9;
  else if (amount > 500) return amount * 0.95;
  else return amount;

Тесты:
1. amount = 1500 → ожидаем 1350 (покрытие ветки >1000)
2. amount = 750 → ожидаем 712.5 (покрытие ветки >500)
3. amount = 300 → ожидаем 300 (покрытие else)
```

---

### ⬛⬜ Gray Box Testing (Тестирование серого ящика)
**Суть:** Частичное знание внутренней структуры. Комбинация black и white box.

**Когда применять:**
- Интеграционное тестирование
- Тестирование API
- Проверка баз данных

**Пример тест-кейса:**
```
ID: GB-001
Название: Проверка кэширования данных
Известно: Данные кэшируются на 5 минут
Шаги:
1. Выполнить запрос GET /api/users/123
2. Зафиксировать время ответа: 250ms
3. Повторить запрос через 1 минуту
4. Ожидаем ответ из кэша: 50ms
5. Повторить через 6 минут
6. Ожидаем полный запрос: 250ms
```

## 2️⃣ ПО УРОВНЯМ ТЕСТИРОВАНИЯ (TESTING PYRAMID)

### Схема пирамиды тестирования:
```
           /\
          /  \       Acceptance Tests
         /____\          (10%)
        /      \
       /        \     Integration Tests
      /__________\        (20%)
     /            \
    /              \   Unit Tests
   /________________\     (70%)
```

---

### 🔹 Unit Testing (Модульное тестирование)
**Суть:** Тестирование отдельных компонентов/функций изолированно.

**Пример тест-кейса:**
```javascript
// Тестируемая функция
function sum(a, b) {
  return a + b;
}

// Тест-кейсы
describe('sum()', () => {
  test('возвращает сумму двух положительных чисел', () => {
    expect(sum(2, 3)).toBe(5);
  });
  
  test('работает с отрицательными числами', () => {
    expect(sum(-1, -1)).toBe(-2);
  });
  
  test('обрабатывает ноль', () => {
    expect(sum(0, 5)).toBe(5);
  });
});
```

---

### 🔹 Integration Testing (Интеграционное тестирование)
**Суть:** Проверка взаимодействия между модулями/сервисами.

**Пример тест-кейса:**
```
ID: INT-001
Название: Интеграция платежной системы с базой данных
Шаги:
1. Создать заказ со статусом "pending"
2. Отправить запрос в платежный шлюз
3. Получить ответ "success" от шлюза
4. Проверить, что статус заказа изменился на "paid"
5. Проверить, что запись о транзакции создана в БД
6. Проверить отправку email пользователю
Ожидаемый результат: Все системы синхронизированы корректно
```

---

### 🔹 System Testing (Системное тестирование)
**Суть:** Тестирование полной интегрированной системы на соответствие требованиям.

**Пример тест-кейса:**
```
ID: SYS-001
Название: Полный цикл оформления заказа
Предусловия: Товар есть на складе
Шаги:
1. Зайти на сайт как незарегистрированный пользователь
2. Найти товар через поиск
3. Добавить в корзину
4. Оформить заказ (ввод данных доставки)
5. Выбрать способ оплаты
6. Оплатить заказ
7. Получить подтверждение на email
8. Проверить статус в личном кабинете
Ожидаемый результат: Заказ создан, оплата прошла, email получен
```

---

### 🔹 Acceptance Testing (Приемочное тестирование)
**Суть:** Финальная проверка перед выпуском. Подтверждение, что система готова к эксплуатации.

**Виды:**
- **UAT (User Acceptance Testing)** - тестирование конечными пользователями
- **BAT (Business Acceptance Testing)** - проверка бизнес-требований
- **Alpha/Beta Testing** - внутреннее/внешнее тестирование

**Пример тест-кейса:**
```
ID: UAT-001
Название: Реалистичный сценарий покупки (от лица пользователя)
Роль: Покупатель
Сценарий:
"Как покупатель, я хочу найти и купить товар, 
чтобы получить его с доставкой на дом"

Критерии приемки:
✓ Поиск работает быстро (<2 сек)
✓ Фильтры применяются корректно
✓ Корзина сохраняется между сессиями
✓ Оплата проходит безопасно
✓ Трек-номер доставки приходит в течение 1 часа
```

## 3️⃣ ПО ЦЕЛЯМ ТЕСТИРОВАНИЯ

### A. Functional Testing (Функциональное)

#### ✅ Positive Testing (Позитивное)
**Суть:** Проверка работы с валидными данными.

```
ID: POS-001
Название: Регистрация с корректными данными
Данные:
  Email: valid@email.com
  Пароль: SecurePass123!
  Имя: Иван
Ожидаемый результат: Успешная регистрация, переход в профиль
```

#### ❌ Negative Testing (Негативное)
**Суть:** Проверка реакции на невалидные данные/действия.

```
ID: NEG-001
Название: Регистрация с уже существующим email
Данные:
  Email: existing@email.com (уже в базе)
  Пароль: AnyPass123
Ожидаемый результат: 
  - Ошибка: "Email уже зарегистрирован"
  - Код ответа: 409 Conflict
  - Запись в БД не создана
```

### B. Non-Functional Testing (Нефункциональное)

#### ⚡ Performance Testing (Тестирование производительности)

**Схема:**

```
     │                     │                      │

```

**Пример тест-кейса:**

## 4️⃣ ПО ВРЕМЕНИ И ЧАСТОТЕ ВЫПОЛНЕНИЯ

### 🚀 Smoke Testing (Дымовое)
**Суть:** Быстрая проверка критического функционала перед глубоким тестированием.

**Схема:**
```
Новая сборка
     │
     ↓
Smoke Tests (5-10 мин)
     │
     ├─→ ❌ Fail → Отклонить сборку, вернуть разработчику
     │
     └─→ ✅ Pass → Запустить полное тестирование
```

**Пример чек-листа:**
```
SMOKE CHECKLIST v1.0
Приложение: Интернет-магазин

[ ] Приложение запускается
[ ] Главная страница загружается
[ ] Поиск работает
[ ] Добавление в корзину
[ ] Авторизация
[ ] Оформление заказа (базовый сценарий)

Время выполнения: 10 минут
Ответственный: QA Engineer
```

---

### 🔄 Regression Testing (Регрессионное)
**Суть:** Проверка, что новые изменения не сломали существующий функционал.

**Стратегии:**
```
Full Regression        Selective Regression      Automated Regression
     │                       │                          │
Все тесты              Только затронутые          Автоматизированные
(раз в спринт)         модули + интеграции        тесты (CI/CD)
     │                       │                          │
Ручное + Авто          Приоритизация по          Nightly builds
                       рискам
```

**Пример тест-кейса:**
```
ID: REG-001
Название: Регресс модуля "Корзина"
Связанные изменения: Изменение расчета скидок

Тесты для проверки:
1. Добавление товара в корзину
2. Изменение количества
3. Удаление товара
4. Применение промокода
5. Расчет итоговой суммы
6. Переход к оформлению

Приоритет: High
Время выполнения: 45 минут
```

---

### 🔁 Retesting (Повторное тестирование)
**Суть:** Проверка конкретных исправленных багов.

**Пример:**
```
ID: RET-001
Связанный баг: BUG-1234
Описание бага: При вводе спецсимволов в поиск приложение падает

Шаги для ретеста:
1. Воспроизвести шаги из бага
2. Ввести: !@#$%^&*()
3. Нажать "Поиск"

Ожидаемый результат: 
  ✓ Приложение не падает
  ✓ Показывается сообщение "Ничего не найдено"
  ✓ Или корректная обработка спецсимволов

Статус: ✅ Fixed / ❌ Not Fixed
```

## 5️⃣ СПЕЦИАЛЬНЫЕ ВИДЫ ТЕСТИРОВАНИЯ

### 🤖 Exploratory Testing (Исследовательское)
**Суть:** Одновременное изучение, тестирование и документирование без заранее написанных тест-кейсов.

**Сессия (Time-boxed):**
```
Session Charter: Исследовать новый функционал "Избранное"

Duration: 90 минут

Focus Areas:
  - Добавление/удаление товаров
  - Синхронизация между устройствами
  - Работа без интернета
  
Notes:
  [10:00] Начал с добавления товара - работает быстро
  [10:15] Обнаружил: при быстром добавлении дублируются товары
  [10:30] Проверил синхронизацию - задержка 2-3 секунды
  [10:45] Оффлайн режим не работает - баг #1235
  
Bugs Found: 3
Coverage: ~70% функционала
```

---

###  Ad-hoc Testing (Случайное)
**Суть:** Неформальное тестирование без плана и документации.

**Пример:**
```
Цель: Найти неочевидные баги перед релизом

Подход:
  - Быстрые случайные действия
  - Нестандартные сценарии
  - "А что если нажать здесь 100 раз?"
  
Примеры действий:
  □ Быстрое нажатие кнопок
  □ Ввод очень длинного текста
  □ Отключение интернета в процессе операции
  □ Изменение времени на устройстве
  □ Одновременное открытие множества вкладок
```

---

### 🌐 A/B Testing
**Суть:** Сравнение двух версий функционала для определения лучшей.

**Схема:**
```
        Трафик (100%)
              │
         ┌────┴────┐
         │         │
    Версия A    Версия B
    (Control)   (Variant)
      50%          50%
         │         │
         └────┬────┘
              │
    Сравнение метрик:
    - Конверсия
    - CTR
    - Время на странице
    - Revenue per user
```

**Пример:**
```
ID: AB-001
Тестируемый элемент: Кнопка "Купить"

Версия A (Control):
  - Цвет: Зеленый
  - Текст: "Купить"
  - Размер: Средний

Версия B (Variant):
  - Цвет: Оранжевый
  - Текст: "Добавить в корзину"
  - Размер: Большой

Метрика успеха: Конверсия в добавление в корзину
Длительность: 2 недели
Статистическая значимость: 95%

Результат: Версия B показала +15% конверсии → Внедряем B
```

## 6️⃣ AUTOMATED vs MANUAL TESTING

### Схема выбора:
```
                    Нужно автоматизировать?
                           │
              ┌────────────┴────────────┐
              │                         │
           ДА (если)                 НЕТ (если)
              │                         │
  ✓ Частое выполнение       ✓ Одноразовые тесты
  ✓ Стабильный функционал   ✓ UX/Usability тесты
  ✓ Критичный функционал    ✓ Exploratory testing
  ✓ Data-driven тесты       ✓ Ad-hoc тесты
  ✓ Regression тесты        ✓ Быстрая проверка гипотез
```

**Пример автоматизированного теста:**
```python
# Selenium WebDriver пример
from selenium import webdriver
from selenium.webdriver.common.by import By

def test_login():
    driver = webdriver.Chrome()
    driver.get("https://example.com/login")
    
    # Тест-кейсы
    driver.find_element(By.ID, "email").send_keys("user@test.com")
    driver.find_element(By.ID, "password").send_keys("password123")
    driver.find_element(By.ID, "login-btn").click()
    
    # Assert
    assert "Dashboard" in driver.title
    assert driver.current_url == "https://example.com/dashboard"
    
    driver.quit()
```

## 📋 СВОДНАЯ ТАБЛИЦА ВИДОВ ТЕСТИРОВАНИЯ

| Вид тестирования | Когда применять | Кто выполняет | Автоматизация |
|------------------|-----------------|---------------|---------------|
| **Unit** | Во время разработки | Разработчики | ✅ 90-100% |
| **Integration** | После объединения модулей | QA + Dev | ✅ 70-80% |
| **System** | Перед релизом | QA | ✅ 50-60% |
| **Acceptance** | Финальная проверка | Product Owner / Users | ❌ 10-20% |
| **Smoke** | Каждая сборка | QA | ✅ 100% |
| **Regression** | После изменений | QA | ✅ 80-90% |
| **Performance** | Перед нагрузкой | QA Perf | ✅ 70% |
| **Security** | Регулярно / перед релизом | Security QA | ✅ 50% |
| **Exploratory** | Когда нет документации | QA | ❌ 0% |
| **Compatibility** | Перед релизом | QA | ⚠️ Частично |

## 💡 BEST PRACTICES

1. **Пирамида тестирования:** Больше unit-тестов, меньше UI-тестов
2. **Shift Left:** Начинайте тестирование как можно раньше
3. **Автоматизируйте разумно:** Не всё нужно автоматизировать
4. **Документируйте:** Даже exploratory тесты нужно фиксировать
5. **Метрики:** Отслеживайте coverage, defect density, escape rate
