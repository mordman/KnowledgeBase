**Change Data Capture (CDC)** — это архитектурный паттерн и набор технологий для отслеживания изменений данных в источнике (обычно в базе данных) и доставки этих изменений в другие системы в реальном времени или близком к реальному.

Простыми словами: CDC фиксирует каждую операцию `INSERT`, `UPDATE`, `DELETE` и передает информацию о том, *какие данные и как* изменились, потребителю (другая БД, Data Lake, Kafka, поисковый движок и т.д.).

### Как это работает (основные подходы)

1. **На основе триггеров (в БД)** — создаются специальные таблицы аудита. Минус: нагрузка на источник и сложность поддержки.
2. **На основе логов транзакций (наиболее популярно)** — CDC-инструмент читает лог изменений (например, WAL в PostgreSQL, binlog в MySQL, CDC-лог в Oracle). Плюс: минимальная нагрузка, работа в реальном времени.
3. **На основе временных меток** — требует колонки `last_modified`, но не видит удаления и физические удаления строк.

## Примеры использования CDC

### 1. Репликация между базами данных в реальном времени
- **Сценарий**: OLTP-база (например, PostgreSQL) → аналитическая OLAP-база (ClickHouse или Vertica).
- **Зачем**: Избежать блокировок на основной БД при тяжёлых аналитических запросах. Данные синхронизируются за миллисекунды.

### 2. Стримовая аналитика (Kafka + Flink/Spark Streaming)
- **Сценарий**: Логи транзакций MySQL → Kafka (CDC коннектор) → Flink/Spark для агрегации → Dashboard (Grafana).
- **Зачем**: Анализировать поступления, средний чек, активность пользователей с задержкой < 1 секунды.

### 3. Обновление поискового индекса (Elasticsearch)
- **Сценарий**: При изменении товара в PostgreSQL → CDC передаёт изменения → Elasticsearch обновляет индекс.
- **Зачем**: Не нужно перестраивать индекс целиком. Поиск по сайту всегда актуален.

### 4. Создание Data Lakehouse / Data Mesh
- **Сценарий**: Все изменения из десятков микросервисных БД → единый Kafka → лендинг в S3 в формате Delta Lake / Iceberg.
- **Зачем**: Единый источник правды для data-команд, возможность воссоздать состояние БД на любой момент времени.

### 5. Инвалидация кэша (Redis, Memcached)
- **Сценарий**: CDC слушает БД, и когда пользователь меняет профиль → удаляет старый кэш или обновляет Redis.
- **Зачем**: Кэш всегда свежий, нет протухания с ошибочными данными.

### 6. Синхронизация между микросервисами (без API-вызовов)
- **Сценарий**: Сервис «Заказы» меняет статус → CDC отправляет событие → Сервис «Доставка» получает.
- **Зачем**: Снижение связанности (event-driven архитектура), гарантия доставки.

## Готовые решения (инструменты)

### Открытые / популярные

| Инструмент | Особенности | Форматы | Поддерживаемые БД (источники) |
|------------|-------------|---------|-------------------------------|
| **Debezium** | Плагин к Kafka Connect, распределённый, отказоустойчивый | Avro, JSON, Protobuf | PostgreSQL, MySQL, MongoDB, SQL Server, Oracle, DB2, Cassandra, Vitess |
| **Airbyte** | EL(T) платформа, хорош для Data Lake, поддерживает CDC | JSON, Parquet, Avro | 300+ коннекторов, но качество CDC разное |
| **Maxwell’s Daemon** | Легковесный, пишет в Kafka, Kinesis, Google Pub/Sub | JSON | Только MySQL / MariaDB |
| **Canal** (Alibaba) | Очень производительный, но сложнее в настройке | Protobuf, JSON | MySQL |
| **Striim** (частично open source) | Оперативная аналитика, трансформации на лету | JSON, XML, CSV | Oracle, SQL Server, Sybase, MySQL, PostgreSQL |
| **PeerDB** | Специализирован на Postgres, быстрая репликация в Data Warehouses | – | PostgreSQL → Snowflake, BigQuery, Redshift |

### Коммерческие / Enterprise

| Решение | Сильные стороны |
|---------|----------------|
| **Oracle GoldenGate** | Де-факто стандарт для Oracle, низкая задержка, двунаправленная репликация |
| **Qlik Replicate** (Attunity) | GUI, поддержка многих БД и Data Lakes, перезагрузка без остановки |
| **AWS DMS** (Database Migration Service) | Умеет CDC после полной загрузки, управляется из консоли AWS |
| **Fivetran** | Автоматическое обнаружение схемы, настройка «из коробки», но дорого |
| **IBM InfoSphere Data Replication** | Для мейнфреймов и DB2 z/OS |

### Встроенный CDC в базах данных

- **PostgreSQL** — логическая репликация + `pgoutput`
- **SQL Server** — Change Tracking (CT) и Change Data Capture (встроенная)
- **MongoDB** — Change Streams (CDC-подобный API)
- **Oracle** — Oracle XStream / GoldenGate in-kernel

## Ключевые требования при внедрении CDC

1. **Idempotent Consumer** — потребители событий должны корректно обрабатывать дубликаты (возможны при перезапуске).
2. **Schema Evolution** — нужно уметь обрабатывать изменения колонок в источнике (например, Debezium c Avro и Schema Registry).
3. **Первичные ключи** — обязательны, чтобы идентифицировать запись для `UPDATE`/`DELETE`.
4. **Задержка** — обычно 100 мс – 5 сек. Ультра-низкая задержка (<5 мс) требует специализированных решений типа Oracle GoldenGate.
5. **Полная загрузка + CDC** — сначала накатываем снэпшот, затем догоняем изменения.

## Краткий итог

CDC — это **не просто репликация**, а технологический фундамент для **реактивных систем**, **аналитики реального времени**, **event-driven архитектур** и **гибридных транзакционно-аналитических (HTAP)** решений.

Если вам нужна надежная, масштабируемая CDC для микросервисов и Kafka — выбирайте **Debezium**. Для простой синхронизации MySQL → BigQuery — **Airbyte** или **Maxwell**. Если ваш мир — Oracle и строгие SLA — **Oracle GoldenGate**.
